# Clinical governance

## Protocol lifecycle

```text
source identification
→ rights review
→ structured draft
→ professional review
→ lived-experience review
→ scenario evaluation
→ multilingual review
→ versioned release
→ post-release monitoring
```

## Required metadata

Every protocol must document:

- author;
- reviewers;
- version;
- sources;
- license;
- intended population;
- indications;
- contraindications;
- known risks;
- supported languages;
- review date;
- next review date.

## Change control

Clinical content changes require a dedicated pull request and must not be
bundled invisibly into infrastructure work.

## Claims

Open-source availability, model fluency, and protocol citations do not establish
clinical effectiveness. Claims must follow evidence.

## Conflicts of interest

Reviewers should disclose relevant commercial, professional, or research
interests.

## Incident learning

The project should define a privacy-preserving process for reporting:

- harmful advice;
- overinterpretation;
- memory errors;
- dependency-promoting language;
- failed crisis handling;
- cultural or translation failures.
